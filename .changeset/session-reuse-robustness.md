Signing in recovers cleanly when your browser's leftover session cookies no longer match the server.

**Affects:** End users, Client app developers, Operators

**End users:** If your browser holds stale or partial device-session cookies from a previous sign-in â€” because the browser evicted one of the pair, you've been idle long enough that the server cleaned up your session, or a previous sign-in was interrupted â€” the next app you open now drops you on the familiar email code form instead of a minimal generic sign-in screen. Your stale cookies are cleared at the same time so the next visit starts from a clean slate and can go straight into the cross-app session-reuse fast path again.

**Client app developers:** No public API or query-parameter changes. Internally, the auth service's `/oauth/authorize` now requires **both** `dev-id` **and** `ses-id` cookies to be present before redirecting the browser into pds-core's upstream session-reuse path; a half-pair is treated as no session and falls through to the email-OTP form. When a half-pair is detected, the email-form response also emits `Set-Cookie: dev-id=; Max-Age=0` and `Set-Cookie: ses-id=; Max-Age=0` clears for both names in both host-only and shared-parent-domain scopes. pds-core additionally mounts a pre-route middleware in front of `/oauth/authorize` and `/account` that intercepts any request resolving to a device with zero bound accounts (stale-pair, migration-purged bindings, fixation race) and 303-redirects to the auth service with `prompt=login` plus the same cookie-clearing `Set-Cookie` headers, so upstream `@atproto/oauth-provider`'s stock three-button welcome page ("Authenticate / Create a new account / Sign in / Cancel") is never rendered from an ePDS entry point. The enriched account chooser gains a visual refinement: when the flow resolves to `epds_handle_mode=random`, handles are hidden from the row text (surfaced as a `title` tooltip instead) so random handles do not become the primary identifier the user sees.

**Operators:** No new configuration. The pre-route guard is mounted automatically when `@atproto/oauth-provider` is initialised. Handle-mode resolution in pds-core's chooser now uses the same three-level precedence as the auth service's signup form (query-string `epds_handle_mode` â†’ client metadata `epds_handle_mode` â†’ `EPDS_DEFAULT_HANDLE_MODE` env var â†’ `picker-with-random`) â€” no behavioural change for existing deployments that leave `epds_handle_mode` unset, but operators who set the env var will see the chooser respect it as well as the signup form. The half-pair cookie clear uses the same shared-parent-domain rule as the existing domain-rewrite middleware (auth hostname ends with `.<PDS_HOSTNAME>` â†’ parent is `PDS_HOSTNAME`; otherwise host-only only), so deployments on unrelated hostnames (e.g. Railway preview envs) silently skip the domain-scoped clear without configuration.
